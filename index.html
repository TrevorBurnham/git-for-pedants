Git’s Three File Trees

  1. Working Tree: All the files currently in your project dir
    • (Except .git/)
  2. Staging Area: All the files that will be in the next commit
    • a.k.a. the index
  3. Head Commit: All the files in the commit you’ve checked out

Copying Files Across Trees

  1. git stage = Working Tree ️➡️ Staging Area
  2. git reset = Head Commit ️➡️ Staging Area
  3. git commit = ️Staging Area ➡️ New Commit

What You Should Really Be Worried About 🐺

Git’s Hash Addiction

  • Hash( Input ): A miracle function! 🙌🏻🤖🙌🏻
  • Every version of a file (“blob”) has a hash
    • Hash( File_Contents )
  • Every version of a directory (“tree”) is recursively hashed
    • Hash( Blobs + File_Names + Trees + Dir_Names )
  • So what is a commit?
    • Root_Tree + Parent_Commit
    • + Message + Author + Committer + Timestamps

Profound Implications

  1. My commit da39a3 is the same as your da39a3.
    • In fact, its entire history is the same!
  2. Branches are dumb.
    • No magic, just a pointer. E.g.: master 👉🏻 da39a3
  3. You can recover any work you commit even if you delete the branch.
    • ✨ git reflog ✨
    • …For 30 days. Then 🚮

Conclusion

All The Handy Tips In One Slide

  • git commit -m? Where’s the fire, buddy? 👨🏻‍🚒
    • Use git commit -v as a time for tranquil reflection. 🌳
  • git merge all those commits? 🐞
    • Use git merge --no-ff to keep them under one umbrella. ☂️
  • git log is a lie. 🎂
    • Use git log --graph to see the whole picture. 📈
  • git rebase leaves author timestamps out of sorts. 🔁
    • Use git rebase --ignore-dates to put them in order. 👌🏻
  • git rebase -i does everything else you could possibly want. 😌
